## 1. 项目背景与目标重定义

### 1.1 当前项目背景

当前项目已经具备首版多人联机闭环：

- 大厅
- 房间
- 对战
- 结算
- 再来一局

现有实现基于：

- 前端：`Next.js + React + TypeScript`
- 数据与认证：`Supabase`
- 实时同步：`Supabase Realtime + RPC Snapshot`
- 对局推进：前端定时触发数据库 RPC

当前真实代码入口主要分布在：

- `src/app/page.tsx`
- `src/app/room/[code]/page.tsx`
- `src/app/battle/[matchId]/page.tsx`
- `src/app/result/[matchId]/page.tsx`
- `src/lib/supabase/game-store.ts`
- `supabase/migrations/20260413103000_add_game_rpcs.sql`

### 1.2 当前核心问题

本次设计不是围绕“局部优化”，而是围绕以下三个根问题展开：

1. 当前产品形态仍然偏网站，而不是移动端游戏。
2. 当前竖屏移动端体验不够纯粹，主视图、信息层级和操作路径不符合游戏产品思路。
3. 当前实时链路仍然属于“轻实时互动网页”，对战体验不够权威、不够直接、不够像真正游戏。

### 1.3 重定义后的目标

本次重构目标不是把现有网页做得更精致，而是把“从进入大厅到打完一局”的主链路，重构成真正具备游戏产品感的移动端竖屏对战体验。

目标闭环固定为：

`大厅 -> 创建/加入 -> 房间编队 -> 实时对战 -> 结算 -> 再来一局`

### 1.4 范围边界

本轮明确只做“核心体验纯游戏化”，不进入以下外围系统：

- 成长系统
- 商城/货币系统
- 排行榜
- 任务/活动体系
- 复杂社交体系
- 长线元游戏循环

本轮的目标是先把“像游戏的核心主链路”做正确，再为未来继续扩展正式游戏产品打底。

## 2. 当前产品与技术现状诊断

### 2.1 产品层问题诊断

#### 2.1.1 首页问题

当前首页仍然是典型 Web 页面结构：

- 昵称输入
- 创建房间卡片
- 加入房间卡片
- 支持模式说明
- 题型范围说明

这导致首页承担了过多“说明页”和“功能页”职责，而没有形成“游戏大厅”的主舞台感。

#### 2.1.2 房间页问题

当前房间页虽然已经实现分队、复制房间码和开始对战，但整体结构仍然更像：

- 成员列表页
- 状态说明页
- 功能操作页

而不是“战前编队准备室”。

#### 2.1.3 战斗页问题

当前战斗页已经具备一定氛围和动画，但本质仍然是：

- React 页面布局
- DOM 卡片
- CSS 动画拼装
- 成员列表长期常驻
- 主战场被信息区和说明区挤压

这会导致战斗页“看起来有战斗元素”，但用户实际感受仍偏网页。

#### 2.1.4 结算页问题

当前结算页更像数据展示页，而不是一局结束后的战报页与复局入口。

### 2.2 技术层问题诊断

#### 2.2.1 实时同步主链路问题

当前实时同步是混合方案：

- `Supabase Realtime` 负责表变更通知
- 页面收到通知后补拉 `snapshot`
- 对战页每 `500ms` 调用 `game_tick_match`

这说明当前架构本质上仍然是：

- 数据库状态为主
- 前端负责推进
- 事件通知只是辅助刷新

这类架构适合 MVP，不适合要求更纯正游戏感的实时对战。

#### 2.2.2 权威状态位置错误

当前对局推进逻辑的重要部分仍由前端持续触发：

- 倒计时推进
- 题目超时
- 切题
- 对局结束判定

这意味着客户端不是纯展示层，而是参与了权威状态推进。这会带来：

- 实时感不够纯粹
- 一致性风险上升
- 断线与重连恢复复杂
- 更难扩展到更强对战节奏

#### 2.2.3 扩展性风险

当前内容首版是“二年级数学对战”。如果后续继续按当前思路直接追加功能，很容易出现以下耦合：

- 学科耦合到页面文案
- 年级耦合到房间配置
- 题型耦合到输入组件
- 判题逻辑耦合到某一类题目

这会阻碍后续扩展：

- 三年级
- 四年级
- 语文选择题
- 英语单词题
- 其他题型或内容包

## 3. 目标产品定位与体验原则

### 3.1 产品定位

本项目定位为：

`以实时对战为核心的移动端竖屏房间派对游戏`

当前内容切入点是：

- 首发内容包：`二年级数学`

但产品本体定位不是“二年级数学网页游戏”，而是：

`先用二年级数学验证的实时对战游戏底座`

### 3.2 体验目标

用户进入产品后的心理感受，应当从“操作网页”切换为“进入一局游戏”。

目标体验节奏：

1. 一进首页，立即感受到游戏大厅氛围。
2. 点击创建/加入后，像执行开局动作，而不是填写表单。
3. 房间页像战前编队，而不是等待页。
4. 战斗页像真正对战界面，而不是信息页。
5. 答题结果立刻转化为攻击反馈。
6. 结算页强化胜负情绪，并缩短复局路径。

### 3.3 体验原则

本次重构遵守以下产品原则：

- 战场/行动优先于说明
- 主舞台优先于卡片列表
- 状态表达优先于长文解释
- 单手竖屏优先于桌面兼容思路
- 核心操作一屏完成
- 能按钮化的不要表单化
- 能游戏化的不要网页化

## 4. 页面级产品体验蓝图

### 4.1 首页大厅

首页定位为“开局大厅”，不是说明页。

首页只保留四类元素：

- 游戏标题与主视觉
- 两个绝对主动作：`创建游戏`、`加入游戏`
- 少量辅助入口：昵称、帮助
- 极轻量一句话状态提示

首页不再长期承载：

- 大段规则说明
- 题型范围说明卡
- 模式说明卡
- 多组信息内容块

### 4.2 创建/加入弹窗

用户已确认：

- 首页采用“战场封面式大厅”
- 点击 `创建游戏 / 加入游戏` 后，使用 `居中弹窗` 承接

因此弹窗设计原则固定为：

- 形式上像“开局面板”，不是网页表单框
- 一屏内完成操作，不依赖滚动
- 主按钮始终清晰
- 模式与容量尽量按钮化
- 房间码输入强识别

### 4.3 房间准备室

用户已确认房间页方向为：

`编队战场式准备室`

因此房间页主职责固定为：

- 展示红蓝两队阵地关系
- 让玩家直观理解站位与人数
- 让玩家直接换边
- 让房主快速发起对战
- 让房间码和邀请能力保持存在，但降到辅助层

房间页不再以：

- 列表
- 状态说明块
- 功能信息卡

作为主要结构。

### 4.4 战斗页

用户已确认战斗页方向为：

`上战场，下操作台`

战斗页结构固定为三层：

1. 顶部战斗状态条
   - 血量
   - 倒计时
   - 题号/阶段
2. 中部战场主视区
   - 阵营
   - 攻击
   - 命中
   - 受击
   - 掉血
3. 底部操作台
   - 当前题目
   - 输入
   - 发射按钮
   - 冷却/结果反馈

成员列表、解释信息和低优先级状态都不应长期占据战斗主视区。

### 4.5 结算页

结算页升级为“战报页 + 再开入口”。

必须保留：

- 胜方主标题
- 红蓝最终血量
- 本局关键数据
- `再来一局` 作为最强按钮
- 返回大厅作为次级动作

结算页目标不是“展示数据”，而是“收尾与快速复局”。

## 5. 核心技术基座设计

### 5.1 总体原则

- 客户端负责展示与输入，不负责推进对局权威状态
- 每个对局必须只有一个权威协调实例
- Supabase 不再承担实时主链路职责
- 核心对局流程必须与题库内容解耦

### 5.2 推荐技术结构

建议最终采用以下结构：

- `Next.js App Shell`
  - 大厅
  - 弹窗
  - 房间页
  - 结算页
  - 基础身份/分享/外围 UI

- `Battle Runtime`
  - 建议升级为 `Canvas/Phaser 战场层 + DOM HUD`
  - 负责真正战斗演出和输入反馈

- `Realtime Match Coordinator`
  - 单房间/单对局权威协调实例
  - 负责题目推进、抢答判定、伤害结算、胜负判定

- `Content Registry`
  - 学科
  - 年级
  - 内容包
  - 难度与规则绑定

- `Answer Evaluator`
  - 统一判题接口

- `Persistence Writer`
  - 事件落库
  - 结果存档
  - 回放与统计

### 5.3 推荐职责切分

#### 游戏层

负责：

- 房间
- 编队
- 开始
- 抢答
- 伤害
- 结算

#### 内容层

负责：

- 学科
- 年级
- 题型
- 内容包
- 难度曲线

#### 判题层

负责：

- 输入协议
- 标准化
- 判题结果

#### 表现层

负责：

- 战场演出
- HUD
- 音效
- 震动
- 局内反馈

### 5.4 对局权威链路

推荐对局链路：

1. 玩家进入房间后连接实时协调层
2. 协调层下发房间权威快照
3. 房主开始后，协调层加载规则包和内容包
4. 协调层生成题目实例并广播
5. 玩家提交答案后，由协调层做权威判定
6. 协调层广播命中、伤害、切题、结束事件
7. 关键结果异步写入 Supabase

### 5.5 内存态与落库策略

推荐采用“内存态实时权威 + 关键节点持久化 + 事件批量写入”：

- 房间成员与准备态：
  - 内存为主
  - 关键动作同步写库
- 对局状态：
  - 内存为主
- 关键节点：
  - 开始
  - 结束
  - 再来一局
  - 离房
  - 立即落库
- 事件：
  - 攻击
  - 掉血
  - 超时
  - 切题
  - 批量写库
- Checkpoint：
  - 周期性保存状态恢复点

目标是：

- 实时性依赖内存权威
- 恢复能力依赖 checkpoint 与事件日志
- 数据库不再承担毫秒级协调职责

### 5.6 推荐实时协调方案

建议优先考虑：

- `Cloudflare Durable Objects` 作为单房间单实例协调单元

推荐理由：

- 天然适合单房间状态权威
- 适合 WebSocket 实时协调
- 状态边界清晰
- 易于实现“一个房间一个实例”

备选方案：

- 自建 Node.js/TypeScript 实时房间服务

不推荐继续沿用的主方案：

- 仅基于 `Supabase Realtime + Postgres Changes + 快照补拉 + 前端 tick`

### 5.7 内容扩展协议

系统必须从第一版开始支持未来扩展：

- 二年级数学
- 三年级数学
- 四年级数学
- 语文选择题
- 英语单词题

为此需要统一抽象：

```ts
type RoomConfig = {
  mode: "1v1" | "1v2" | "1v3" | "2v2" | "3v3";
  subject: "math" | "chinese" | "english";
  grade: "g2" | "g3" | "g4";
  contentPackId: string;
  rulesetId: string;
};

type QuestionInstance = {
  id: string;
  prompt: string;
  inputSchema: "single_number" | "quotient_remainder" | "single_choice";
  evaluatorId: string;
  difficulty: number;
  damage: number;
  timeLimitMs: number;
};
```

关键点：

- 客户端按 `inputSchema` 渲染输入控件
- 服务端按 `evaluatorId` 执行判题
- 游戏核心不关心题目到底属于数学、语文还是英语

## 6. 可扩展内容体系设计

### 6.1 内容体系目标

首发内容是：

- 二年级数学

但系统底座必须原生支持未来升级为：

- 多年级
- 多学科
- 多题型
- 多内容包

### 6.2 推荐扩展模型

建议采用：

- `Subject`：学科
- `Grade`：年级
- `Content Pack`：内容包
- `Question Template`：题目模板
- `Evaluator`：判题器
- `Input Schema`：输入协议

### 6.3 扩展方式

未来扩内容时，应尽量做到：

- 新增三年级数学：新增内容包
- 新增英语单选：新增题型渲染器 + 判题器 + 内容包
- 新增语文选择题：新增内容包与对应 evaluator

不应要求：

- 改大厅主流程
- 改房间主流程
- 改战斗主流程
- 改实时协调主流程

## 7. 里程碑拆分与实施顺序

### 7.1 里程碑原则

每个里程碑都必须满足：

- 更像游戏，不像网站
- 真机竖屏可用
- 可验证
- 不依赖后续大重构兜底

### 7.2 推荐里程碑

#### M1：游戏化外壳重建

目标：

- 首页大厅重做
- 创建/加入弹窗重做
- 房间准备室重做
- 统一竖屏 UI 规范与动效规范

#### M2：战斗表现层重建

目标：

- 战斗视图重构为 `上战场，下操作台`
- 战斗主舞台权重提升
- 战斗反馈统一进入节奏化体验

#### M3：实时对局基座重建

目标：

- 引入实时协调层
- 把抢答、切题、伤害、结束判定从前端迁出
- 建立内存权威 + 持久化分层结构

#### M4：结算与持续迭代能力

目标：

- 结算战报化
- 再来一局流程稳定
- 回放/战报基础
- 联机验证与异常监控能力

### 7.3 实施顺序建议

推荐顺序：

1. 先锁产品形态
2. 再重建战斗视图
3. 再引入实时协调层
4. 最后收口结算与验证

不建议顺序：

- 先做强实时服务，再回头补产品形态

原因：

- 如果页面职责与产品体验没定型，后端能力会被错误前台浪费

## 8. 风险、边界与验收标准

### 8.1 核心风险

- 范围失控
- UI 游戏化但底层仍是网页交互
- 实时链路双权威冲突
- 首个内容包把底座写死
- 移动端键盘与输入体验翻车

### 8.2 M1 验收标准

- 首页一眼被感知为游戏大厅
- 创建/加入弹窗单屏完成
- 房间页具备编队准备室体验
- 常见竖屏尺寸下无关键操作遮挡

### 8.3 M2 验收标准

- 战斗页形成清晰主视区
- 成员信息不再长期侵占战斗区
- 命中与答题反馈进入统一时序体验
- 用户能明显感知“答题立刻引发战斗动作”

### 8.4 M3 验收标准

- 对局主推进不再依赖前端持续 tick
- 协调层成为唯一权威
- 断线可恢复
- 多局连续对战状态稳定

### 8.5 M4 验收标准

- 结算页战报化完成
- 再来一局多人流程稳定
- 存在基本事件回放或战报能力
- 建立双设备、弱网、长连验证能力

## 9. 推荐落地路径与下一阶段计划

### 9.1 推荐总体路径

本项目推荐正式进入：

`纯游戏化核心体验重建计划`

其落地路径为：

- 外壳仍保留 `Next.js`
- 战斗层升级为真正游戏运行时
- 实时层升级为权威协调服务

### 9.2 下一阶段直接产出物

下一阶段不应立即“整体大改代码”，而应先产出一份实施计划，明确：

- 仓库结构调整方案
- 技术选型落地顺序
- 每个里程碑的文件级改造范围
- 最小可验证版本
- 测试与验收策略

### 9.3 最终结论

本次重构的本质不是视觉升级，而是以下三件事同时成立：

1. 主链路产品形态从网站改成游戏
2. 战斗页从网页拼装改成真正战斗界面
3. 实时主链路从数据库驱动改成权威协调驱动

只有这三件事同时成立，项目才会从“可玩的联机原型”升级为“真正具备游戏产品感的核心体验底座”。
